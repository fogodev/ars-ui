## Spec vs. issue discrepancy (RESOLVED ŌĆö spec wins per CLAUDE.md "spec is the contract")

The issue's "Tests to add first" lists two tests that contradict the spec:

- ŌØī `` `label` prop produces `aria-label` ``
- ŌØī `` `label_id` prop produces `aria-labelledby` ``

The spec's `Props` struct (┬¦1.1) has exactly six fields ŌĆö `id`, `disabled`, `invalid`,
`read_only`, `role`, `dir` ŌĆö and **no** `label`/`label_id`. Spec ┬¦3.1 explicitly marks

**Decision:** Implement per spec, drop the two contradicting test bullets, replace with

## Internal spec inconsistency ŌĆö resolved on accessibility merits

Spec ┬¦3.1 prose said: "When `role` is `Presentation`, state attributes (`aria-disabled`,
`aria-invalid`) are omitted since the grouping has no semantic meaning."

Spec ┬¦1.2 code example: unconditionally emits state attributes regardless of `role`.

**Best-outcome decision: always emit state attributes when the prop is true, regardless of
role.** Reasoning:

1. **WAI-ARIA 1.2 ┬¦5.4** explicitly states `aria-*` global states and properties ARE still
   exposed on elements with `role="presentation"` ŌĆö the role only suppresses the *implicit*
   role mapping, not state attributes. The original ┬¦3.1 prose was based on a misreading.
2. **React Aria parity** (┬¦5.1 documents full parity): React Aria emits state attributes
   unconditionally on its `Group`. Diverging here would silently break parity that the spec
   commits to.
3. **`GroupContext` is the load-bearing propagation mechanism**, not root ARIA. The root
   `aria-disabled`/`aria-invalid`/`aria-readonly` are decorative complements that help AT
   when the disabled state visually applies to the whole group; gating them on `role`
   creates asymmetric behavior between the propagation path (always works) and the AT
   announcement (sometimes works).
4. **`data-ars-*` styling hooks must always emit** regardless of role so CSS targeting stays
   consistent. Splitting aria-* vs data-* on role-gated logic introduces a footgun.
5. **Simpler mental model**: `disabled=true ŌćÆ aria-disabled on root`, end of story. Fewer
   branches, fewer edge cases, lower test surface.

**Implementation consequence:** Match the ┬¦1.2 code example. **Spec consequence:** Replace
the ┬¦3.1 line "When `role` is `Presentation`, state attributes (`aria-disabled`,
`aria-invalid`) are omitted..." with a brief note clarifying that state attributes are
always emitted (and citing the React Aria parity / WAI-ARIA ┬¦5.4 justification). Lands in
the same PR per CLAUDE.md "no spec drift" rule.

# Plan: Lighter local `cargo xci` (`--profile fast`)

## Context

`cargo xci` is the local mirror of `.github/workflows/ci.yml`'s 20 gates. We just measured it
taking ~25 minutes locally ŌĆö the bulk of that is the `coverage` step (~15 minutes; wasm32
`-Cinstrument-coverage` recompile of every adapter crate via nightly + LLVM 22) and the
five `feature-matrix-*` steps (~5 minutes combined). After a normal implementation flow the
relevant unit / integration / adapter test suites have already run during TDD iteration and
the `post-implementation-audit` skill; re-running them under coverage measurement before push
is pure redundancy. Five different feature-matrix combos checking subsystem crates the change
didn't touch is also overkill for a local gate whose only job is "catch what focused tests
miss before this push lands on origin".

We need an opt-in **fast profile** that covers cross-crate / workspace-level regressions
without paying the wasm-coverage and feature-matrix taxes. Target run time: ~5 minutes on a
warm cache.

## Step selection

Per the Phase-1 audit of `xtask/src/ci/mod.rs` lines 30ŌĆō125, the 20 steps split cleanly:

**Fast profile ŌĆö 10 steps, ~5 min:**

| Step | Why it stays |
|---|---|
| `Fmt` | Catches formatter drift across the workspace; ~15 s. |
| `Check` | Type-error gate; fast and foundational. |
| `Clippy` | `-D warnings` gate; required for merge. |
| `Unit` | Workspace-wide nextest run; catches inter-crate regressions focused tests miss. |
| `Integration` | Black-box suite; separate from unit, fast enough, catches real bugs. |
| `Adapter` | Leptos + Dioxus harness; touches both adapters in one pass. |
| `SpecCompileSnippets` | Catches spec drift Codex can't see; fast. |
| `ErrorVariantCoverage` | Catches untested `ComponentError` variants; fast lint. |
| `MutualExclusion` | The `icu4x` + `web-intl` compile-guard; ~3 s and load-bearing. |
| `SnapshotCount` | Per-component snapshot policy lint; ~3 s. |

**Skipped from fast ŌĆö 10 steps, ~20 min saved:**

| Step | Why it's skipped from fast (still runs in `cargo xci` full) |
|---|---|
| `I18nBrowser`, `DomBrowser` | wasm-pack browser runs, only relevant when ars-i18n / ars-dom code changes; skip-by-default and re-add positionally when you touched those crates. |
| `Release` | Release-profile compile/test; release-only failures are rare. |
| `AdapterParity` | Per-component test-count diff lint; useful but not pre-push critical. |
| `Coverage` | The big one ŌĆö wasm `-Cinstrument-coverage` recompile + nightly + LLVM 22. The same tests already ran in `Unit`/`Integration`/`Adapter`; redundant pre-push. CI still enforces thresholds. |
| `FeatureMatrixCore`, `FeatureMatrixI18n`, `FeatureMatrixSubsystems`, `FeatureMatrixLeptos`, `FeatureMatrixDioxus` | Feature-combination CI concern; rarely catches anything a workspace-wide `--all-features` `check`/`clippy`/`unit` pass didn't already. |

Power users who touched wasm/feature code can opt the skipped steps back in positionally:
`cargo xci --profile fast i18n-browser dom-browser`. Positional steps override the profile
(preserves the current `cargo xci fmt check` selection semantics).
