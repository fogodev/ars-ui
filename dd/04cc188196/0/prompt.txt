## Spec vs. issue discrepancy (RESOLVED ŌĆö spec wins per CLAUDE.md "spec is the contract")

The issue's "Tests to add first" lists two tests that contradict the spec:

- ŌØī `` `label` prop produces `aria-label` ``
- ŌØī `` `label_id` prop produces `aria-labelledby` ``

The spec's `Props` struct (┬¦1.1) has exactly six fields ŌĆö `id`, `disabled`, `invalid`,
`read_only`, `role`, `dir` ŌĆö and **no** `label`/`label_id`. Spec ┬¦3.1 explicitly marks
`aria-label` and `aria-labelledby` as "Consumer-provided", meaning adapter users supply them
via prop spreading, not via Group props. Inventing `label`/`label_id` props would silently
diverge from the React Aria parity table (┬¦5.1) which already shows full parity without them.

**Decision:** Implement per spec, drop the two contradicting test bullets, replace with
spec-aligned tests below.

## Internal spec inconsistency ŌĆö resolved on accessibility merits

Spec ┬¦3.1 prose said: "When `role` is `Presentation`, state attributes (`aria-disabled`,
`aria-invalid`) are omitted since the grouping has no semantic meaning."

Spec ┬¦1.2 code example: unconditionally emits state attributes regardless of `role`.

**Best-outcome decision: always emit state attributes when the prop is true, regardless of
role.** Reasoning:

1. **WAI-ARIA 1.2 ┬¦5.4** explicitly states `aria-*` global states and properties ARE still
   exposed on elements with `role="presentation"` ŌĆö the role only suppresses the *implicit*
   role mapping, not state attributes. The original ┬¦3.1 prose was based on a misreading.
2. **React Aria parity** (┬¦5.1 documents full parity): React Aria emits state attributes
   unconditionally on its `Group`. Diverging here would silently break parity that the spec
   commits to.
3. **`GroupContext` is the load-bearing propagation mechanism**, not root ARIA. The root
   `aria-disabled`/`aria-invalid`/`aria-readonly` are decorative complements that help AT
   when the disabled state visually applies to the whole group; gating them on `role`
   creates asymmetric behavior between the propagation path (always works) and the AT
   announcement (sometimes works).
4. **`data-ars-*` styling hooks must always emit** regardless of role so CSS targeting stays
   consistent. Splitting aria-* vs data-* on role-gated logic introduces a footgun.
5. **Simpler mental model**: `disabled=true ŌćÆ aria-disabled on root`, end of story. Fewer
   branches, fewer edge cases, lower test surface.

**Implementation consequence:** Match the ┬¦1.2 code example. **Spec consequence:** Replace
the ┬¦3.1 line "When `role` is `Presentation`, state attributes (`aria-disabled`,
`aria-invalid`) are omitted..." with a brief note clarifying that state attributes are
always emitted (and citing the React Aria parity / WAI-ARIA ┬¦5.4 justification). Lands in
the same PR per CLAUDE.md "no spec drift" rule.

## Implementation outline

Match `spec/components/utility/group.md` ┬¦1.1 and ┬¦1.2 line-for-line. Specifically:

1. **`GroupRole`** ŌĆö `#[derive(Clone, Copy, Debug, PartialEq, Eq, Default)]` enum with three
   variants (`Group` default, `Region`, `Presentation`). Use `#[default]` on `Group`.

2. **`Props`** ŌĆö `#[derive(Clone, Debug, Default, PartialEq, HasId)]`. The spec writes a manual
   `impl Default` that produces identical output to derive; use derive for DRY (matches
   `Separator`'s convention). Builder chain matching separator's pattern: `new()`, `id()`,
   `disabled()`, `invalid()`, `read_only()`, `role()`, `dir()`. Setters that don't take
   `String` are `const fn`.

3. **`GroupContext`** ŌĆö `#[derive(Clone, Debug, PartialEq)]` plain struct. Three `pub bool`
   fields exactly as spec ┬¦1.2. Named `GroupContext` (not `Context`) per foundation 08 ┬¦8.4
   ("Secondary contexts or helpers use descriptive non-prefixed names ŌĆö `GroupContext`,
   `QueueContext`, ..."), so consumers write `group::GroupContext`.

4. **`Part`** ŌĆö `#[derive(ComponentPart)] #[scope = "group"] pub enum Part { Root }`.

5. **`Api`** ŌĆö `#[derive(Clone, Debug)] pub struct Api { props: Props }`. Methods:
   `new(Props) -> Self` (`const`), `props() -> &Props`, `id() -> &str`,
   `group_context() -> GroupContext`, `root_attrs() -> AttrMap`. `root_attrs` follows the
   spec code: set Id, scope/part data attrs, then a `match` on role emitting
   `role="group"|"region"|"presentation"`, then three independent `if` blocks emitting
   `aria-disabled` + `data-ars-disabled`, `aria-invalid` + `data-ars-invalid`,
   `aria-readonly` + `data-ars-readonly`, then optional `dir`.

6. **`ConnectApi for Api`** ŌĆö single-arm `match part { Part::Root => self.root_attrs() }`.
