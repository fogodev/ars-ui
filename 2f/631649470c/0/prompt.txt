Why this matters: these three utilities are foundational structural primitives (semantic headings, page landmarks, text-match highlight). Without them, downstream components that compose Heading/Landmark/Highlight (Dialog titles, Toast regions, Combobox highlighting, AlertDialog headings) keep accruing TODOs in their adapter specs. Shipping them now unblocks several of the open stateless-utility adapter tasks (#322 Group, #323 Field/Fieldset, #325 keyboard, etc.).

**Per user decision:** one combined PR (closes all four issues), both SSR + wasm tests, and ship `HeadingLevelProvider` + `Section` alongside `Heading` so the spec's "nested heading-level inheritance must flow through published `HeadingContext`" invariant is testable end-to-end in this PR.

## Scope

For each adapter (Leptos + Dioxus), implement:

1. **`heading`** module
   - `Heading` component — renders `<h1>` … `<h6>` based on resolved level (props override > context > default Level::One). Native heading elements only — no fallback `<div role="heading">` since we can always pick an h-tag.
   - `HeadingLevelProvider` component — provider-only, publishes a starting `HeadingContext` via framework context.
   - `Section` component — provider-only, publishes the incremented `HeadingContext` for descendants.
2. **`landmark`** module
   - `Landmark` component — renders semantic element per `Role`: `<header>` / `<nav>` / `<main>` / `<aside>` / `<footer>` / `<form>` / `<section>` for non-Search roles, and `<div role="search">` fallback for `Search` (matches `Api::prefers_generic_fallback_element()`). Localized `aria-label` resolved via `use_locale()` + adapter-passed `messages`.
3. **`highlight`** module
   - `Highlight` component — renders `<span>` root + iterates `Api::chunks(&locale)`, emitting `<mark>` for highlighted chunks and `<span>` for unmatched. Locale resolved from optional `locale` prop or `use_locale()` provider.

No `AsChild` variants in this PR — the adapter specs do not call for them on these three utilities (the agnostic Props have no `as_child` field), and the canonical implementation sketches in each spec render the semantic element directly without a render callback.

## Patterns Reused (Do Not Reinvent)

- `crate::attr_map_to_leptos_inline_attrs` / `crate::attr_map_to_dioxus_inline_attrs` — convert `AttrMap` to spreadable framework attrs. (Already used by `separator.rs` and `visually_hidden.rs` in both adapters.)
- `crate::use_locale()` — read the locale from `ArsContext`. Both adapters expose this via `provider.rs`.
- `provide_context(...)` (Leptos) / `use_context_provider(|| ...)` (Dioxus) — publish `HeadingContext` for descendants.
- `use_context::<HeadingContext>()` (Leptos) / `try_consume_context::<HeadingContext>()` (Dioxus) — read inherited level, defaulting to `HeadingContext::new()` (Level::One) when absent.
- `leptos::either::{Either, EitherOf6, EitherOf8}` — fan-out for dynamic Leptos element tags (h1-h6 for Heading, six native + fallback `<div>` for Landmark).
- `ars_components::utility::{heading, landmark, highlight}::{Api, Part, Props, …}` — agnostic core; re-export from the adapter module so end-users can write `heading::Level::Three`, `landmark::Role::Navigation`, `highlight::MatchStrategy::Contains`, etc.
- Existing test recipes — Leptos uses `view! { … }.to_html()` for SSR, Dioxus uses `VirtualDom::new(app) + rebuild_in_place() + dioxus_ssr::render(&vdom)` (copy the helper from `tests/visually_hidden.rs`). Wasm tests use `wasm_bindgen_test_configure!(run_in_browser);` and either the existing `ars-test-harness-leptos::render(…)` / `ars-test-harness-dioxus::render(…)` for mount/state, or — for stateless utilities — direct DOM assertions on a manually mounted node.

## Component-by-Component Implementation

### 1. `ars-leptos` — `heading` module

Re-export the agnostic surface and provide three components:

```rust
pub use ars_components::utility::heading::{
    Api, HeadingContext, Level, Part, Props,
    heading_level_provider, section,
};

#[component]
pub fn Heading(
    #[prop(optional, into)] id: Option<Oco<'static, str>>,
    #[prop(optional)] level: Option<Level>,
    children: Children,
) -> impl IntoView {
    let ctx = use_context::<HeadingContext>().unwrap_or_default();
    let mut props = Props::new();
    if let Some(id) = id.as_ref() {
        props = props.id(id.to_string());
    }
    if let Some(level) = level {
        props = props.level(level);
    }
    let api = Api::new(props, ctx);
    let resolved = api.resolved_level();
    let attrs = attr_map_to_leptos_inline_attrs(api.root_attrs(true));

    match resolved {
        Level::One   => EitherOf6::A(view! { <h1 {..attrs}>{children()}</h1> }),
        Level::Two   => EitherOf6::B(view! { <h2 {..attrs}>{children()}</h2> }),
        Level::Three => EitherOf6::C(view! { <h3 {..attrs}>{children()}</h3> }),
        Level::Four  => EitherOf6::D(view! { <h4 {..attrs}>{children()}</h4> }),
        Level::Five  => EitherOf6::E(view! { <h5 {..attrs}>{children()}</h5> }),
        Level::Six   => EitherOf6::F(view! { <h6 {..attrs}>{children()}</h6> }),
    }
}

#[component]
pub fn HeadingLevelProvider(level: Level, children: Children) -> impl IntoView {
    let provider_props = heading_level_provider::Props::new().level(level);
    let ctx = heading_level_provider::context_for(&provider_props);
    provide_context(ctx);
    children()
}

#[component]
pub fn Section(children: Children) -> impl IntoView {
    let parent = use_context::<HeadingContext>().unwrap_or_default();
    provide_context(section::context_for(&parent));
    children()
}
```

Notes:
- Heading always passes `is_native_heading_element = true` because we can always emit a real `<h{1..6}>` — there is no need for the `role="heading" aria-level=N` fallback path in a Leptos web adapter.
- When the user provides no `id`, the core emits `id=""`. Strip the empty id before converting to attrs (mirrors how `separator.rs::root_attrs` handles `id`).

### 2. `ars-leptos` — `landmark` module

```rust
pub use ars_components::utility::landmark::{Api, Messages, Part, Props, Role};

#[component]
pub fn Landmark(
    #[prop(optional, into)] id: Option<Oco<'static, str>>,
    #[prop(optional)] role: Option<Role>,
    #[prop(optional, into)] labelledby_id: Option<Oco<'static, str>>,
    #[prop(optional, into)] messages: Option<Messages>,
    children: Children,
) -> impl IntoView {
    let resolved_role = role.unwrap_or(Role::Region);
    let mut props = Props::new().role(resolved_role);
    if let Some(id) = id.as_ref() { props = props.id(id.to_string()); }
    if let Some(lb) = labelledby_id.as_ref() {
        props = props.labelledby_id(lb.to_string());
    }
    let messages = messages.unwrap_or_default();
    let env = ars_core::Env { locale: use_locale().get(), ..ars_core::Env::default() };
    let api = Api::new(props, &env, &messages);
    let is_native = !api.prefers_generic_fallback_element();
    let attrs = attr_map_to_leptos_inline_attrs(api.root_attrs(is_native));

    if !is_native {
        Either::Left(view! { <div {..attrs}>{children()}</div> })
    } else {
        Either::Right(match resolved_role {
            Role::Banner        => EitherOf7::A(view! { <header  {..attrs}>{children()}</header> }),
            Role::Navigation    => EitherOf7::B(view! { <nav     {..attrs}>{children()}</nav> }),
            Role::Main          => EitherOf7::C(view! { <main    {..attrs}>{children()}</main> }),
            Role::Complementary => EitherOf7::D(view! { <aside   {..attrs}>{children()}</aside> }),
            Role::ContentInfo   => EitherOf7::E(view! { <footer  {..attrs}>{children()}</footer> }),
            Role::Form          => EitherOf7::F(view! { <form    {..attrs}>{children()}</form> }),
            Role::Region        => EitherOf7::G(view! { <section {..attrs}>{children()}</section> }),
            Role::Search        => unreachable!("Search role is handled by prefers_generic_fallback_element()"),
        })
    }
}
```

Notes:
- Locale comes from `use_locale()`. We need the resolved `Locale` value (not the signal) to construct `Env`. The agnostic API stores it eagerly, so non-reactive read is acceptable for the initial implementation. (Locale changes mid-life will not retrigger label resolution — that matches the agnostic-core contract: `Api::new` snapshots the locale at construction.)
- `messages` defaults to `Messages::default()` (empty label). For the spec's "Form and Region require accessible name" path, callers pass `Messages { label: MessageFn::static_str("…") }` or wire `use_messages()`.
- `is_native_landmark_element` is `false` only when `Api::prefers_generic_fallback_element()` is true (currently only `Role::Search`).

### 3. `ars-leptos` — `highlight` module

```rust
pub use ars_components::utility::highlight::{
    Api, HighlightChunk, MatchStrategy, Part, Props,
};

#[component]
pub fn Highlight(
    #[prop(optional, into)] query: Vec<String>,
    #[prop(optional, into)] text: String,
    #[prop(optional)] ignore_case: Option<bool>,
    #[prop(optional)] match_strategy: Option<MatchStrategy>,
    #[prop(optional)] locale: Option<Locale>,
) -> impl IntoView {
    let props = Props::new()
        .query(query)
        .text(text)
        .ignore_case(ignore_case.unwrap_or(true))
        .match_strategy(match_strategy.unwrap_or_default());
    let api = Api::new(props);
    let locale = locale.unwrap_or_else(|| use_locale().get());

    let root_attrs = attr_map_to_leptos_inline_attrs(api.root_attrs());
    let chunks = api.chunks(&locale)
        .into_iter()
        .map(|chunk| {
            let chunk_attrs = attr_map_to_leptos_inline_attrs(api.chunk_attrs(chunk.highlighted));
            let text = chunk.text.to_string();
            if chunk.highlighted {
                Either::Left(view! { <mark {..chunk_attrs}>{text}</mark> })
            } else {
                Either::Right(view! { <span {..chunk_attrs}>{text}</span> })
            }
        })
        .collect_view();

    view! { <span {..root_attrs}>{chunks}</span> }
}
```

Notes:
- `api.chunks(&locale)` returns chunks borrowing from `props.text`. We own `props` inside the call so the chunks live long enough; copy each `chunk.text` into an owned `String` before constructing the view.
- The root `dir="auto"` comes from `api.root_attrs()` already — do not duplicate.

### 4-6. `ars-dioxus` — Same three modules

Dioxus is structurally identical, with these differences:

- Use **explicit `#[derive(Props, Clone, PartialEq, Debug)]` structs** for every component (workspace convention; see `feedback_dioxus_explicit_props.md`).
- Use **`try_use_context::<HeadingContext>()`** to read inherited context (returns `Option<HeadingContext>`; matches the convention used throughout `crates/ars-dioxus/src/provider.rs`, `nonce.rs`, and `platform.rs`).
- Use `use_context_provider(|| ctx)` for publishing context.
- Match arms return `rsx! { … }` directly — Dioxus's `Element` is the universal return type, no `EitherOf*` needed.
- Read locale via the existing `crate::use_locale()` helper. Both `ars_leptos::use_locale()` and `ars_dioxus::use_locale()` return `Signal<Locale>`. Read the current value with `.get()` (Leptos) or `.read().clone()` (Dioxus) — see `crates/ars-dioxus/src/provider.rs:418` for the canonical "use prop if present, else snapshot from signal" pattern.

Heading skeleton:

```rust
#[derive(Props, Clone, PartialEq, Debug)]
pub struct HeadingProps {
    #[props(optional, into)] pub id: Option<String>,
    #[props(optional)]       pub level: Option<Level>,
    pub children: Element,
}

#[component]
pub fn Heading(props: HeadingProps) -> Element {
    let ctx = try_use_context::<HeadingContext>().unwrap_or_default();
    let mut core_props = Props::new();
    if let Some(id) = props.id.as_deref() { core_props = core_props.id(id); }
    if let Some(level) = props.level     { core_props = core_props.level(level); }
    let api = Api::new(core_props, ctx);
    let attrs = attr_map_to_dioxus_inline_attrs(api.root_attrs(true));
    let children = props.children;

    match api.resolved_level() {
        Level::One   => rsx! { h1 { ..attrs, {children} } },
        Level::Two   => rsx! { h2 { ..attrs, {children} } },
        Level::Three => rsx! { h3 { ..attrs, {children} } },
        Level::Four  => rsx! { h4 { ..attrs, {children} } },
        Level::Five  => rsx! { h5 { ..attrs, {children} } },
        Level::Six   => rsx! { h6 { ..attrs, {children} } },
    }
}

#[derive(Props, Clone, PartialEq, Debug)]
pub struct HeadingLevelProviderProps {
    pub level: Level,
    pub children: Element,
}

#[component]
pub fn HeadingLevelProvider(props: HeadingLevelProviderProps) -> Element {
    let provider_props = heading_level_provider::Props::new().level(props.level);
    use_context_provider(|| heading_level_provider::context_for(&provider_props));
    rsx! { {props.children} }
}

#[derive(Props, Clone, PartialEq, Debug)]
pub struct SectionProps { pub children: Element }

#[component]
pub fn Section(props: SectionProps) -> Element {
    let parent = try_use_context::<HeadingContext>().unwrap_or_default();
    use_context_provider(|| section::context_for(&parent));
    rsx! { {props.children} }
}
```

## Risks / Edge Cases to Watch

- **Locale signal mid-life updates** — Leptos `use_locale()` returns a signal; `Landmark::Api::new(props, env, messages)` snapshots the locale once. If the user changes locale post-mount, the rendered `aria-label` will not update unless we move the `Api` construction into a `Memo`. For this PR's scope (stateless utility, parity with the canonical sketches), one-shot construction is correct per spec §7 "render-time only, no post-mount sync expected". If a downstream component needs reactive locale, we revisit then — not now.
- **Heading context default** — When `use_context::<HeadingContext>()` returns `None`, default to `HeadingContext::new()` (Level::One). Do NOT panic.
- **Highlight chunk borrow** — `Api::chunks(&locale)` returns chunks borrowing from `props.text`. The current `props` lives inside the function; clone each `chunk.text` to an owned `String` before constructing the view (cheap — most chunks are short and we are not in a hot path).
- **Dioxus rsx returns under `match`** — Dioxus 0.7 supports returning `rsx! { … }` from each arm directly as `Element`. No need for unifying types.
- **`leptos::either::EitherOf*` availability** — Leptos 0.8 ships `Either`, `EitherOf3`…`EitherOfN` up to `EitherOf16`. If a specific arity is missing during implementation, use the next higher size (e.g., `EitherOf8` with one unused variant, marked `unreachable!()`) rather than `AnyView`. Confirm imports via the **leptos** skill before writing code that depends on these types — claudes' training data on Leptos APIs is stale.
- **Locale read pattern** — both `ars_leptos::use_locale()` and `ars_dioxus::use_locale()` return `Signal<Locale>` (verified). Leptos reads with `.get()`; Dioxus reads with `.read().clone()`. Use the canonical "use prop if present, else snapshot from signal" pattern from `crates/ars-dioxus/src/provider.rs:418` so locale-prop overrides work uniformly.
- **Search role** — `<search>` is intentionally NOT used; the agnostic core's `prefers_generic_fallback_element()` already signals this; we honor it by rendering `<div role="search">`.
- **No snapshot files expected** — These adapters render plain HTML attribute strings, so `.to_html()` and `dioxus_ssr::render` assertions use direct string matching. No `insta` fixtures should be created. If any `.snap` files appear in the diff, re-examine — they likely indicate accidental adoption of snapshot tests in adapter crates (those belong in the agnostic core).
