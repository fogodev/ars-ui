# Plan: Implement FocusScope agnostic core (task #212)

## Context

Task #212 (epic #10) implements the **framework-agnostic FocusScope core** — a stateful utility that constrains keyboard Tab focus within a container, enabling focus trapping for `Dialog`, `Drawer`, `Popover`, and other overlay components. The spec at `spec/components/utility/focus-scope.md` defines its contract; the issue lists 10 tests and the acceptance criteria.

Why this matters: FocusScope is critical infrastructure. Several already-implemented components (Dialog, Popover) currently model focus-trap behavior inline via per-component `Effect::FocusInitial` / `Effect::FocusFirstTabbable` markers. Landing FocusScope's agnostic core gives those overlays a reusable trap primitive and unblocks downstream work (Drawer, ComboBox listbox, etc.).

The spec at §1.6 contains drift: its transition closures call `use_platform_effects()` synchronously, but that function lives in adapter crates (`ars-leptos`, `ars-dioxus`) — the agnostic core cannot reach it. Dialog and Popover have already established the correct pattern: a typed `Effect` enum with `PendingEffect::named(Effect::Foo)` markers that adapters dispatch on (`crates/ars-components/src/overlay/dialog/mod.rs:549`, `crates/ars-components/src/overlay/popover/mod.rs:96`). This task adopts that pattern for FocusScope and updates the spec to match.

## State machine transitions

```text
(Inactive, Activate { trapped, saved_focus_id })
  → Active { trapped: trapped || props.contain }
  apply: ctx.saved_focus = saved_focus_id
  effect: PendingEffect::named(Effect::FocusTrapListener)
  then(Event::FocusFirst) if props.auto_focus

(Active { .. }, Deactivate { restore_focus })
  → Inactive
  cancel_effect: Effect::FocusTrapListener  (adapter cleanup tears down keydown handler)
  if restore_focus:
    effect: PendingEffect::named(Effect::RestoreFocus)  (adapter reads ctx.saved_focus)
  else:
    apply: ctx.saved_focus = None

(Active { trapped: false }, TrapFocus) → Active { trapped: true }
(Active { trapped: true },  ReleaseTrap) → Active { trapped: false }

(Inactive, RestoreFocus)
  → Inactive (no state change)
  effect: PendingEffect::named(Effect::RestoreFocus)

(Active { .. }, RestoreFocus) → None  (ignored — only meaningful after Deactivate)

(Active { .. }, FocusFirst) → Active (no state change), effect: PendingEffect::named(Effect::FocusFirst)
(Active { .. }, FocusLast)  → Active (no state change), effect: PendingEffect::named(Effect::FocusLast)
(Inactive, FocusFirst | FocusLast) → None

_ → None
```

`Machine::initial_effects` keeps the trait default (returns empty `Vec`) — the spec has no `default_active` prop so `init` always returns `State::Inactive`. No initial open-lifecycle needs to fire.

## Connect API (`container_attrs`)

```rust
let mut attrs = AttrMap::new();
let [(scope_attr, scope_val), (part_attr, part_val)] = Part::Container.data_attrs();
attrs.set(scope_attr, scope_val).set(part_attr, part_val);
if self.is_active() {
    attrs.set_bool(HtmlAttr::Data("ars-active"), true);
    attrs.set(HtmlAttr::TabIndex, "-1");
}
if self.is_trapped() {
    attrs.set_bool(HtmlAttr::Data("ars-trapped"), true);
}
attrs
```

`Api::is_active()` → `matches!(state, State::Active { .. })`. `Api::is_trapped()` → `matches!(state, State::Active { trapped: true })`. Imperative methods `activate(trapped, saved_focus_id)`, `deactivate(restore_focus)`, `focus_first()`, `focus_last()` all send their corresponding `Event`. `focus_next(_)` and `focus_previous(_)` return `false` (matches spec stub — the adapter exposes platform-aware variants on its own API surface).

`get_tabbable_elements()` is **removed** from the core Api — synchronous DOM queries cannot run in the agnostic core (the issue's "Element/ref handling note" is explicit). Adapters call `PlatformEffects::tabbable_element_ids` directly when consumers need this.

**Issue/spec attribute typo:** Issue #212 test 9 says `data-ars-focus-trapped`, but spec §2 says `data-ars-trapped`. The spec is authoritative — implement `data-ars-trapped`. Cross-reference both names in the PR description.

## Effect contract (adapter handles)

| Effect              | When emitted                                              | Adapter contract                                                                                                              |
| ------------------- | --------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| `FocusTrapListener` | `Inactive → Active` (Activate)                            | `platform.attach_focus_trap(container_id, on_escape) -> CleanupFn`. Wire the returned cleanup as the effect's cleanup.        |
| `FocusFirst`        | `Active + FocusFirst` (also on Activate when auto_focus)  | `platform.focus_first_tabbable(container_id)`. Fall back to focusing the container element when no tabbables exist.           |
| `FocusLast`         | `Active + FocusLast`                                      | `platform.focus_last_tabbable(container_id)`. Fall back to the container.                                                     |
| `RestoreFocus`      | `Active → Inactive` (Deactivate) or explicit RestoreFocus | Read `service.context().saved_focus`. If `Some(id)`, run the §7 fallback chain via `PlatformEffects` methods. Adapters MAY clear via a follow-up event but the core does not require it (next `Activate` overwrites). |

`cancel_effect(Effect::FocusTrapListener)` invokes the cleanup captured at install time. No separate `Effect::FocusTrapRelease` variant is needed.
